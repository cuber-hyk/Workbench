# Workbench 项目描述（供 UI 绘制 Agent 使用）

## 文档目的

本文用于向 UI 绘制 Agent 描述 Workbench 的产品背景、核心对象、业务模块、用户任务、状态语义和使用边界。

本文只描述“这个产品是什么、用户在做什么、系统管理哪些对象、对象之间如何关联”。本文不提供 UI 布局方案、视觉风格建议、组件选择建议或交互实现方案，避免限制后续 UI 设计发挥。

## 产品概述

Workbench 是一个本地优先的 AI 开发工作台。它运行在用户自己的电脑上，用于集中管理本地开发项目、Agent Skills、AI 相关资源记录，以及与本机开发环境相关的自动化入口。

Workbench 的核心定位不是在线协作平台，也不是营销型产品展示页，而是面向个人开发者、AI 工具重度用户、独立研究者和本地工作流管理者的桌面软件。用户希望通过 Workbench 把分散在本机不同目录、不同 Agent 工具、不同项目和不同资料来源中的信息集中起来，减少重复查找、重复配置和手工同步。

Workbench 当前关注四类核心任务：

- 管理本地开发项目。
- 管理和同步 AI Agent Skills。
- 维护 AI 资源 Radar。
- 管理本地设置、路径、工具目录和应用行为。

Workbench 以本地数据为核心。数据主要保存在用户本机 SQLite 数据库中，Skills 文件保存在本机统一根目录中，项目目录、工具目录和资源链接也都以本地可控的方式管理。Workbench 不假设云端账号、不依赖在线数据库，也不默认上传用户项目内容。

## 目标用户

Workbench 的主要用户是熟悉本地开发环境的人，通常具备以下特征：

- 经常在多个本地项目之间切换。
- 使用 Codex、Claude Code、OpenCode 等一个或多个 AI Agent 工具。
- 会维护自己的 Skills、prompts、规则、工具目录或自动化脚本。
- 会收藏 GitHub 项目、技术文章、论文、工具仓库和 AI 相关资料。
- 关心本地文件安全、目录边界、版本冲突和可恢复性。
- 希望 UI 清晰、稳定、可长期高频使用，而不是偏展示或娱乐。

用户在 Workbench 中通常不是第一次接触开发工具的新手。用户更关心信息是否完整、状态是否明确、操作是否可控、错误是否可理解，以及 Workbench 是否会意外修改或覆盖本机文件。

## 产品边界

Workbench 是本地工作台，不是代码托管平台。

Workbench 管理项目记录和启动配置，但不替代 IDE、编辑器或终端。

Workbench 可以打开项目目录、使用外部工具打开项目、启动非交互式命令并展示当前运行会话，但不提供完整交互式 shell。

Workbench 管理 Skills 的真实副本、启用关系和同步目标，但不自动合并冲突文件，不静默覆盖用户已有内容。

Workbench 维护资源 Radar 数据，但不负责后台抓取、定时采集、自动推荐、LLM 总结或评分。

Workbench 支持从 GitHub Stars 同步资源，但同步结果仍然进入本地数据库，由用户维护分类、领域、标签、备注和收藏状态。

Workbench 的重要原则是：任何可能覆盖、删除、接管或改变用户本机文件结构的行为，都必须让用户明确知道发生了什么。

## 本地优先原则

Workbench 的数据和文件都以本地为中心。

核心数据保存在本机 SQLite 数据库中。数据库记录包括项目、启动项、Skills 分类、Skills 启用关系、自定义工具目标、资源 Radar 条目、应用设置等。

Skills 的统一真实来源保存在本机 Workbench Skills 根目录中。外部 Agent 工具目录不是真实来源，它们只是启用目标或发现来源。

项目目录仍然是用户自己的本地目录。Workbench 只保存路径和元数据，不接管项目源码，不移动项目文件，不删除项目目录。

资源 Radar 保存的是本地资源记录。资源链接可能指向 GitHub、文档站、论文页面或其他外部地址，但 Workbench 中的分类、标签、备注和收藏状态由本地数据库维护。

## 核心概念

### 项目

项目是用户本机的一个开发目录。Workbench 保存项目名称、路径、标签、备注、启动项、最近启动状态等信息。

项目记录只是 Workbench 对本地目录的索引。删除项目记录不会删除本地项目目录。

项目可以配置一个或多个启动项。启动项包含名称、命令、工作目录、是否启用、排序等信息。

项目可以通过外部工具打开，例如 VS Code、Trae、PowerShell、Claude Code 等。外部工具打开方式是全局配置，不属于某一个项目。

### 启动项

启动项用于从 Workbench 发起一个非交互式命令。启动命令通常用于启动本地服务、开发服务器、脚本或工具进程。

一个项目可以有多个启动项。项目启动主操作会执行所有启用且配置完整的启动项。

Workbench 会管理当前应用运行期间的启动会话和内存日志。日志不持久化，应用重启后历史日志不会保留。

启动项必须有有效工作目录。工作目录不存在时应阻止启动，并告知用户原因。

### 启动会话

启动会话是一次项目启动产生的运行记录。它可能包含一个或多个启动项的输出。

启动会话状态来自当前 Workbench 应用进程内存，不代表历史运行记录。

用户可以查看本次启动的输出、状态和错误，但 Workbench 不提供完整交互式终端能力。

### Skill

Skill 是一个包含 `SKILL.md` 的目录，用于描述某类 Agent 能力、规则、工作流或工具使用方法。

Workbench 扫描统一 Skills 根目录下的直接子目录，识别其中的 `SKILL.md`，读取 Skill 名称、描述、路径和元信息。

Skill 可以被分类。分类只影响 Workbench 内部整理，不改变 Skill 文件目录。

Skill 可以启用到不同 Agent 工具的全局 Skills 目录，也可以在支持项目级 Skills 的工具中启用到具体项目目录。

### Skills 统一根目录

Skills 统一根目录是 Workbench 管理的唯一真实 Skill 副本位置。

Workbench 的目标是让用户只维护一份 Skill 内容，然后按需启用到多个 Agent 工具目录。

统一根目录中的 Skill 是真实来源。外部工具目录中的同名 Skill 可能是符号链接、复制副本、历史残留内容或冲突来源。

切换统一根目录不会自动迁移旧目录内容，也不会自动重建所有已启用目标。迁移和重建都需要用户显式操作。

### Agent 工具目标

Agent 工具目标是某个 AI Agent 工具的 Skills 目录。例如 Codex、Claude Code、OpenCode 等。

部分工具支持全局 Skills 目录，也支持项目级 Skills 目录。部分工具只支持全局目录。

Workbench 内置若干工具目标，也允许用户添加自定义工具目标。

自定义工具只支持全局 Skills 目录，不支持项目级启用。

### 全局启用

全局启用指把某个 Skill 启用到某个 Agent 工具的全局 Skills 目录。

启用方式可能是符号链接，也可能是复制副本。Workbench 默认优先使用符号链接，失败时使用复制。

全局启用状态需要表达该 Skill 是否已被某个工具可见，以及该目标是否由 Workbench 管理。

### 项目级启用

项目级启用指把某个 Skill 启用到某个项目目录下某个 Agent 工具的项目级 Skills 目录。

项目级启用和全局启用不同，它同时依赖 Skill、项目和工具三个对象。

只有支持项目级 Skills 的工具才允许项目级启用。当前重要工具包括 Codex、Claude Code、OpenCode 等。其他工具可能只支持全局启用。

项目级启用涉及更多状态，例如目标目录缺失、目标已有非 Workbench 管理内容、符号链接失效、复制副本可能过期、项目路径不可访问等。

### Skill 来源

Skill 来源用于记录某个 Skill 是如何进入 Workbench 的。

来源可能包括本地文件夹导入、ZIP 导入、GitHub 仓库导入、skills.sh 市场安装、外部工具目录同步等。

远程来源 Skill 可以记录来源类型、仓库地址、路径、安装引用、内容 hash、最近检查时间等信息，用于后续检查更新。

不是所有 Skill 都有远程来源。用户本地创建或导入的 Skill 也可以没有远程更新能力。

### Skills 市场

Skills 市场用于浏览和安装来自外部来源的 Skill。目前重点来源是 `skills.sh`。

市场中的 Skill 安装后会进入 Workbench 统一 Skills 根目录。

市场安装完成后默认不自动启用到任何 Agent 工具目录。

已安装、未安装、可更新、不支持、安装中、检查失败等状态需要明确区分。

### Skill 更新

Skill 更新用于检查由 Workbench 管理的远程来源 Skill 是否有新版本。

可更新来源包括 skills.sh 和部分 GitHub 分支来源。固定 tag 或固定 commit 导入的 Skill 通常不参与更新检查。

更新执行前需要备份旧版本。更新替换的是统一 Skills 根目录中的真实副本，不应自动覆盖用户未确认的外部工具目录内容。

### 资源 Radar

资源 Radar 是用户维护 AI 相关资源的本地资料库。

资源可以是项目、资讯、论文或其他类型。

资源可以有名称、分类、领域、来源、链接、标签、语言、GitHub Stars、GitHub Topics、来源描述、备注、收藏状态、更新时间等字段。

资源 Radar 的目标是帮助用户记录和整理值得关注的技术资源，而不是自动推荐内容。

### GitHub Stars 同步

GitHub Stars 同步会读取用户当前 GitHub 账号收藏的仓库，并写入或更新本地资源记录。

同步不会覆盖用户手工维护的分类、领域、标签、备注和收藏状态。

如果 GitHub Star 对应的 URL 与已有手动资源匹配，Workbench 可以把 GitHub 来源挂到该资源上。

如果同一个 GitHub 来源匹配多条手动资源，Workbench 不自动合并，而是创建待处理重复组。

### Radar 重复组

重复组用于处理多个资源记录指向同一外部来源的情况。

用户需要选择一个主资源。Workbench 会合并来源、标签、备注和收藏状态，并删除或归并副资源记录。

重复组合并必须由用户明确触发。

### 设置

设置模块用于管理 Workbench 的本地路径、工具目录、项目打开方式、自定义工具、本地数据位置、主题和应用行为。

设置不是业务内容的主入口，而是用于维护 Workbench 运行所需的本地环境配置。

## 主要模块

### 项目模块

项目模块用于管理本地开发项目。

用户可以添加项目、编辑项目、删除项目记录、搜索项目、按标签筛选项目、查看项目详情、打开项目目录、使用外部工具打开项目、配置启动项、启动项目命令、查看当前启动日志。

项目模块中的对象主要包括项目、启动项、启动会话、项目打开方式。

项目模块的重要边界：

- 删除项目记录不删除本地项目目录。
- 项目路径不存在时，需要提示用户路径可能已移动或删除。
- 启动项工作目录不存在时，不允许启动。
- 启动日志只保存当前应用运行期间的内存记录。
- 外部工具打开项目不受 Workbench 进程管理。

### Skills 模块

Skills 模块用于集中管理 Workbench Skills。

用户可以扫描统一根目录、查看本地 Skills、搜索和筛选 Skill、管理分类、启用或停用全局工具目标、管理项目级启用、导入 Skill、从 GitHub 导入 Skill、同步外部工具目录中的 Skills、浏览技能市场、安装市场 Skill、卸载已安装 Skill、检查和执行来源更新、处理冲突。

Skills 模块中的对象主要包括 Skill、分类、统一根目录、工具目标、全局启用、项目级启用、来源记录、市场条目、更新条目、冲突状态。

Skills 模块的重要边界：

- 统一根目录是唯一真实来源。
- 外部工具目录只作为发现来源或启用目标。
- 未被 Workbench 管理的外部目录内容不能被静默覆盖。
- 同名 Skill 内容冲突必须显式处理。
- 删除 Skill 只删除统一根目录中的 Skill 和 Workbench 管理的启用目标，不删除未被管理的外部工具目录内容。
- 市场安装后默认不启用到任何工具。
- 更新前必须备份旧版本。

### 资源 Radar 模块

资源 Radar 模块用于维护本地资源库。

用户可以新增资源、编辑资源、删除资源、搜索资源、按分类筛选、按领域筛选、按来源筛选、收藏资源、打开链接、同步 GitHub Stars、处理来源失效状态、处理重复组。

资源 Radar 中的对象主要包括资源条目、来源、标签、领域、收藏状态、GitHub 元数据、重复组。

资源 Radar 的重要边界：

- Workbench 不自动抓取互联网内容。
- Workbench 不对资源做自动评分或推荐。
- GitHub Stars 同步不覆盖用户维护字段。
- 来源失效不等于删除资源。
- 重复合并必须由用户确认。

### 设置模块

设置模块用于维护 Workbench 本地运行配置。

用户可以查看数据库位置、修改统一 Skills 根目录、查看和管理工具目录、添加自定义 Agent 工具、管理项目打开方式、设置应用行为、管理主题、检查旧根目录迁移状态、重建受管启用目标。

设置模块的重要边界：

- 修改统一 Skills 根目录不会自动迁移旧内容。
- 重建启用目标必须由用户显式触发。
- 删除自定义工具只删除 Workbench 配置，不删除外部工具目录。
- 打开不存在的目录时，需要由用户决定是否创建。

## 关键工作流

### 添加本地项目

用户选择或输入项目路径，填写项目名称、标签、备注和可选启动项。Workbench 保存项目记录。项目路径应唯一。

用户后续可以从项目记录打开目录、使用外部工具打开项目、启动项目命令。

### 启动项目

用户在项目中配置一个或多个启动项。每个启动项包含命令、工作目录和启用状态。

用户执行项目启动后，Workbench 校验所有启用启动项。校验通过后，Workbench 创建当前启动会话，执行命令并捕获输出。

用户可以查看本次启动会话的日志。日志不保存到历史记录。

### 扫描 Skills

Workbench 读取统一 Skills 根目录，扫描直接子目录中的 `SKILL.md`。

扫描结果和数据库中的分类、启用关系、来源记录合并，形成用户可管理的 Skills 列表。

### 导入 Skills

用户可以从 ZIP、文件夹或 GitHub 仓库导入 Skill。

导入单位是包含 `SKILL.md` 的目录整体。

同名冲突需要跳过或由用户确认处理。导入后默认不启用到任何 Agent 工具。

### 同步外部工具目录 Skills

Workbench 可以只读扫描已注册 Agent 工具的全局 Skills 目录，发现外部已有 Skill。

发现阶段不写数据库、不复制文件、不创建目录。

如果发现新增 Skill、不可读目录、不可用路径或内容冲突，需要用户确认后才能导入、接管或处理。

### 全局启用 Skill

用户选择某个 Skill 和某个 Agent 工具，Workbench 将统一根目录中的 Skill 启用到该工具全局 Skills 目录。

Workbench 需要记录目标路径、同步方式和管理状态。

停用时只移除 Workbench 管理的目标，不删除未被管理的外部内容。

### 项目级启用 Skill

用户选择某个 Skill、某个项目和某个支持项目级 Skills 的工具。

Workbench 将该 Skill 启用到项目目录下对应工具的项目级 Skills 目录。

项目级启用需要检查项目路径、工具支持能力、目标目录状态、已有内容、同步方式和冲突状态。

### 处理 Skill 冲突

冲突通常发生在统一根目录和外部工具目录存在同名但内容不同的 Skill。

用户需要选择唯一版本来源。Workbench 在替换前备份被替换版本。

冲突处理不自动合并文件内容。

### 安装市场 Skill

用户在市场中选择 Skill 安装。Workbench 使用隔离临时环境提取 Skill，并复制到统一 Skills 根目录。

安装完成后写入来源记录。Skill 不自动启用到工具目录。

### 检查和更新 Skill 来源

Workbench 根据来源记录检查远端内容是否变化。

如果远端内容变化，用户可以执行更新。更新前备份本地旧版本。

更新完成后刷新本地 Skills 状态和来源记录。

### 添加 Radar 资源

用户手动添加资源名称、分类、领域、链接、标签、备注等信息。

资源进入本地数据库，可被搜索、筛选、收藏和编辑。

### 同步 GitHub Stars

Workbench 通过用户本机 GitHub CLI 账号读取 starred repositories。

同步时根据来源 ID 和规范化 URL 匹配已有资源。

新增资源、更新来源字段、标记来源失效和发现重复组都应有清晰结果。

### 合并 Radar 重复组

用户在重复组中选择主资源。

Workbench 合并来源、标签、备注和收藏状态，保留主资源的主要分类和领域。

副资源只在用户确认合并后处理。

## 状态语义

Workbench 中存在大量状态。状态必须表达含义，而不只是颜色。

### 项目状态

项目可能处于以下状态：

- 未配置启动项。
- 可启动。
- 运行中。
- 部分运行。
- 启动失败。
- 已停止。
- 已结束。
- 项目路径不存在。

### 启动项状态

启动项可能处于以下状态：

- 未配置命令。
- 已启用。
- 已停用。
- 工作目录缺失。
- 正在运行。
- 已结束。
- 启动失败。

### Skill 状态

Skill 可能处于以下状态：

- 正常。
- 未分类。
- 来源可更新。
- 来源不支持更新。
- 来源检查失败。
- 内容冲突。
- 已安装。
- 未安装。
- 安装中。
- 卸载中。

### 全局启用状态

全局启用可能处于以下状态：

- 未启用。
- 已启用。
- Workbench 管理。
- 内容一致但尚未登记。
- 内容冲突。
- 目标缺失。
- 目标不可读。
- 工具目录不可用。

### 项目级启用状态

项目级启用可能处于以下状态：

- 未启用。
- 已启用。
- 已启用且使用 symlink。
- 已启用且使用 copy。
- 目标缺失。
- 内容冲突。
- 工具不支持项目级。
- 项目路径不可访问。
- 复制副本可能过期。
- 目标存在但非 Workbench 管理。

### Radar 资源状态

Radar 资源可能处于以下状态：

- 普通资源。
- 已收藏。
- GitHub 来源有效。
- GitHub 来源失效。
- 待合并重复组。
- 手动添加。
- 同步新增。
- 同步更新。

## 数据对象关系

项目与启动项是一对多关系。一个项目可以有多个启动项。

项目与项目打开方式不是从属关系。项目打开方式是全局配置，可作用于多个项目。

Skill 与分类是多对一关系。一个 Skill 只能属于一个分类，一个分类可以包含多个 Skills。

Skill 与工具目标通过启用关系关联。一个 Skill 可以启用到多个工具，一个工具可以启用多个 Skills。

Skill、项目、工具目标共同决定项目级启用关系。项目级启用不是单纯 Skill 属性，也不是单纯项目属性。

Skill 与来源记录是一对零或一关系。不是所有 Skill 都有远程来源。

Radar 资源与来源可以是一对多关系。一个资源可能同时有手动来源和 GitHub 来源。

重复组包含多个候选资源，用户选择其中一个作为主资源。

## 文案和语气

Workbench 默认使用中文界面。

技术名词、路径、命令、工具名、API 名称、文件名可以保留英文。

文案应具体说明发生了什么，而不是使用泛泛的提示。

错误文案应说明原因和可操作方向。

危险操作文案应明确说明影响范围，例如“只删除 Workbench 记录，不删除本地目录”。

同步、导入、更新、删除、接管等操作应避免模糊词。用户需要知道 Workbench 会读取什么、复制什么、写入什么、删除什么、备份什么。

## 安全和副作用边界

Workbench 涉及大量本机文件操作，因此副作用边界非常重要。

以下操作不应静默发生：

- 覆盖外部工具目录中的已有 Skill。
- 删除未被 Workbench 管理的文件或目录。
- 自动合并内容冲突。
- 自动迁移旧 Skills 根目录。
- 自动重建所有启用目标。
- 自动删除本地项目目录。
- 自动启用市场安装的 Skill。
- 自动覆盖用户维护的 Radar 分类、标签、备注或收藏状态。

以下操作应有明确反馈：

- 扫描完成。
- 导入完成。
- 同步完成。
- 安装完成。
- 更新完成。
- 删除完成。
- 发现冲突。
- 发现不可读路径。
- 发现目标缺失。
- 发现重复资源。

## 给 UI 绘制 Agent 的内容注意事项

Workbench 是一个长期使用的本地开发工作台。它承载的是项目、Skills、资源和本机环境配置，不是营销型页面。

Workbench 的对象信息通常较长，例如 Windows 路径、命令、GitHub URL、工具目录、Skill 描述等。界面内容需要考虑长文本、长路径、长命令和多状态并存的情况。

Workbench 中的状态非常重要。用户需要判断某个对象是否正常、是否启用、是否冲突、是否由 Workbench 管理、是否需要处理。

Workbench 中的许多操作有本机文件副作用。任何涉及删除、覆盖、接管、迁移、重建、同步、更新的操作，都需要让用户清楚知道影响范围。

Workbench 的多个模块之间有联系，但每个模块也有清晰边界。项目模块关注本地项目和启动；Skills 模块关注 Skill 文件、工具启用和来源；Radar 模块关注资源记录；设置模块关注本机路径和配置。

UI 设计时应理解这些业务关系，但本文不规定具体布局、组件形式或视觉风格。
