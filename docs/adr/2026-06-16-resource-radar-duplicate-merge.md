# ADR: 资源 Radar 重复组合并后删除副资源

## Context

资源 Radar 同时支持手动添加资源和 GitHub Stars 同步。二者可能指向同一个 GitHub repo。如果保留多条并行记录，后续筛选、收藏、备注和来源状态会分裂，用户也很难判断哪条才是主记录。

多条手动资源匹配同一个 GitHub repo 时，系统不能安全自动选择主资源，因为每条手动资源可能有不同分类、领域、标签和备注。

## Decision

当 GitHub Stars URL 唯一匹配一条手动资源时，系统自动把 GitHub 来源挂到该手动资源，不创建新资源。

当同一个 GitHub repo 匹配多条手动资源时，系统创建打开状态重复组，由用户选择一个主资源。合并后：

- 主资源保留自己的资源类型和领域。
- 来源列表合并，并写入 GitHub 来源描述和元数据。
- 用户标签去重合并。
- 副资源备注追加到主资源备注后。
- 收藏状态取 OR。
- 副资源删除。
- 重复组标记为 `resolved`。

名称相似但 URL 不同不触发自动合并。

## Alternatives

- 保留所有重复资源并只标记来源冲突：实现简单，但资源身份长期分裂，筛选和详情会持续噪声。
- 自动选择最近更新或收藏的资源作为主资源：省交互，但会替用户做分类和领域归属决策。
- 合并后保留副资源为隐藏记录：降低不可逆风险，但引入双轨状态和恢复语义，超过当前 MVP 复杂度。

## Consequences

- 资源 Radar 中同一个真实资源只有一个主记录，来源和用户维护内容集中展示。
- 用户必须在多候选场景明确选择主资源，系统不替用户决定分类和领域。
- 合并删除副资源是不可逆操作；测试必须覆盖标签、备注、收藏、分类、领域和重复组状态。
