# Workbench PRD

## 1. 产品概述

Workbench 是一个本地优先的 AI 开发工作台，用于统一管理个人开发项目、Skills、资源 Radar 和本机 AI 开发环境入口。

它不是目录型工作流，也不是现有工具的插件，而是一个独立桌面软件。参考项目只作为功能、交互和工程经验来源，不合并代码，不拼成 monorepo。

## 2. 产品目标

Workbench 已完成第一阶段基础能力验证，当前进入进阶开发阶段。

当前目标是把 Workbench 从“可用的本地工具集合”推进为“稳定的个人 AI 开发操作台”：

- 让用户清楚知道本机项目、Skills、Agent 工具和资源 Radar 的真实状态。
- 降低在项目目录、终端、多个 Agent 配置目录和资料来源之间切换的成本。
- 让 Skills 的来源、启用、冲突、同步和更新过程可见、可控、可恢复。
- 让项目启动、项目资料、项目 Skills 和外部工具入口形成闭环。
- 在保持本地优先的前提下，逐步接入可验证的外部来源、导入来源和自动化入口。

## 3. 用户画像

目标用户是本机长期使用 AI 辅助开发的个人开发者。

用户通常同时维护多个本地项目，并使用 Codex、Claude Code、OpenCode 和其他 Agent 工具。用户希望把项目入口、启动方式、Skills、资源和常用操作集中管理，而不是分散在文件夹、终端历史、浏览器收藏夹和多个工具配置目录中。

## 4. 产品原则

- 本地优先：数据默认保存在本机。
- 独立软件：不作为现有项目或工具的插件。
- 简单优先：新能力必须有清晰入口、清晰数据所有权和清晰回退方式。
- 单一实现：不保留多套并行实现，不为历史项目做兼容层。
- 统一来源：Workbench Skills 根目录保存 Skill 的唯一真实副本，工具和项目目录通过 Auto 同步启用。
- 真实状态优先：界面展示应尽量反映本机文件系统、工具目录和数据库的实际状态。
- 可恢复：覆盖、替换、删除和批量操作必须有预览、确认或备份。
- 渐进增强：先把本地闭环做稳，再扩展在线来源、自动化、备份和生态能力。

## 5. 当前能力基线

### 5.1 项目管理

用户可以添加本地项目路径，并在 Workbench 中维护项目基本信息。

已实现能力：

- 添加本地项目路径。
- 从 GitHub 或 Gitee 仓库克隆项目到用户选择的本地父目录。
- 通过系统目录选择器填充项目路径和启动工作目录。
- 展示项目名称、路径、备注、标签。
- 搜索项目。
- 查看项目详情。
- 打开项目目录。
- 删除 Workbench 项目记录。
- 为项目配置一个或多个启动项。
- 每个启动项包含名称、启动命令、启动工作目录和启用状态。
- 通过启动按钮为所有启用启动项创建 Workbench 内嵌启动会话。
- 查看本次启动日志详情，并按“全部”或单个启动项切换输出。
- 停止单个启动会话或停止本次启动的全部会话。
- 配置全局项目打开方式 Profiles，并用外部工具打开项目。

边界：

- 删除项目只影响 Workbench 本地项目记录，不删除、移动或修改本地项目目录。
- 运行中的项目不可删除，需要先停止本次启动会话。
- 访问项目目录或启动工作目录时发现路径不存在，应提示用户是否删除 Workbench 项目记录。
- 远程导入不覆盖已有本地目录；导入前同时检查 Workbench 记录和目标目录。
- 相同路径的记录和目录都存在时直接定位已有项目；记录存在但目录缺失时允许确认重新克隆并保留原项目配置；只有未受管目录存在时要求更换父目录。
- 远程导入失败时显示明确失败状态和错误信息，不停留在运行中进度。
- 当前启动会话日志只保留在应用运行期间，不作为历史日志持久化。

### 5.2 Skills 管理

Workbench 使用统一 Skills 根目录保存所有 Skills。该目录是 Skills 内容的唯一真实来源。

已实现能力：

- 配置 Workbench Skills 根目录。
- 扫描统一根目录下的 `SKILL.md` 文件。
- 解析 Skill 名称、描述和路径。
- 展示 Skills 列表。
- 搜索 Skills。
- 查看 Skill 详情。
- 打开统一 Skills 根目录。
- 打开对应 `SKILL.md` 文件。
- 为 Skill 选择全局启用工具。
- 在 Skills 列表中通过工具标识快速切换全局启用。
- 删除 Skill，删除前必须确认。
- 为 Skill 选择项目及项目内启用工具。
- 对单个项目下当前 Skill 的所有工具进行全部启用或全部关闭。
- 创建和移除由 Workbench 管理的符号链接。
- Auto 同步：优先创建符号链接，创建失败时自动复制到目标目录。
- 记录每个启用项实际使用的 Symlink 或 Copy 方式。
- 目标位置已有内容时不覆盖、不合并。
- 停用时只移除 Workbench 记录的符号链接或完整副本。
- 扫描全局工具目录中的同名 Skill。
- 全局工具目录中同名 Skill 与 Workbench 根目录内容一致时，自动登记为 Workbench 管理。
- 全局工具目录中同名 Skill 与 Workbench 根目录内容不一致时，标记为冲突。
- 冲突由用户在 `.workbench` 和所有存在版本的全局工具目录中选择一个唯一版本源解决。
- 解决冲突前备份被替换版本到 `~/.workbench/backups/skills/<timestamp>/<tool>/<skill>`。
- 应用冲突解决后，选中的版本成为 Workbench 根目录中的唯一真实副本，已存在的全局工具目录统一重新同步。
- 删除 Skill 时删除统一根目录中的 Skill、分类记录和 Workbench 管理的启用记录。
- 删除 Skill 时只清理 Workbench 管理的符号链接或副本，不删除未被 Workbench 管理的工具目录内容。
- 支持自定义 Skill 分类。
- 支持新增、重命名、删除和合并 Skill 分类；删除或合并分类时只迁移 Skills，不删除 Skill 文件。
- 每个 Skill 只能属于一个分类。
- 按分类、启用状态、启用工具和启用项目筛选 Skills。
- 工具筛选同时包含全局启用和项目级启用。
- 全局启用支持 Codex、Claude Code、OpenCode、DevEco Code、Hermes、Kimi Code、Pi Agent、Gemini CLI、Qwen Code、Goose、Kilo Code、Cline、Roo Code、Factory Droid、Amp、Kiro CLI 和 Junie CLI。
- 用户可以新增、编辑和删除自定义 Agent 工具；自定义工具只支持全局 Skills 目录，不支持项目级启用。
- 自定义 Agent 工具只要求用户填写名称、全局 Skills 目录和可选图标；内部标识由 Workbench 自动生成。
- 自定义 Agent 工具名称不能与已有工具重复。
- 自定义工具图标由用户选择本地图标文件，保存到 `~/.workbench/tool-icons/`。
- 项目级启用当前只支持 Codex、Claude Code 和 OpenCode；其他工具先作为全局 Skills 目标管理。
- Skills 列表中的全局工具列按用户设置顺序展示前 4 个工具和 `+N` 浮层。
- Skills 列表在默认窗口宽度下应隐藏可见滚动条，并避免产生横向滚动条；桌面应用默认窗口宽度应为 Skills 表格操作列预留足够空间。
- 在 Skills 列表中通过分类下拉选择已有分类或创建新分类。
- 未分类 Skill 统一归入“未分类”。
- “未分类”是系统分类，始终存在且不可删除。
- 分类只用于 Workbench 内整理，不影响文件目录和符号链接结构。
- 从 ZIP 文件或已解压文件夹导入一个或多个 Skills。
- 导入后的 Skills 默认不启用。
- 若统一根目录已存在同名 Skill，则停止导入该 Skill 并提示冲突，不覆盖、不合并。

### 5.3 资源 Radar

资源 Radar 是本地资源条目库，用于记录项目、资讯、论文和其他信息。资源可以手动添加，也可以从当前 `gh` CLI 账号手动同步 GitHub Stars。

已实现能力：

- 添加资源 Radar 条目。
- 编辑和删除资源 Radar 条目。
- 手动同步当前 `gh` CLI 账号的 GitHub Stars。
- 展示条目列表。
- 搜索条目。
- 按分类、领域、来源、语言、来源状态和重复状态筛选。
- 收藏条目。
- 查看条目详情。
- 打开条目链接。
- GitHub Stars URL 唯一匹配手动资源时合并来源，不新建重复条目。
- 多条手动资源匹配同一个 GitHub repo 时展示待合并重复组，由用户选择主资源后合并。
- 取消 Star 只标记来源失效，不删除本地资源。

分类固定为：

- 项目
- 资讯
- 论文
- 其他

领域 `domain` 用于组织技术方向，例如 `Skills`、`Agent`、`RAG`、`AI 基础`、`开发工具`、`文档工具`、`算法与数据结构`、`教程与资源`、`前端开发`、`Android 开发`、`桌面应用`、`音视频工具`、`安全与网络` 和 `其他`。

### 5.4 设置

设置模块用于管理 Workbench 的基础配置。

已实现能力：

- 查看本地数据存储位置。
- 打开 Workbench 根目录、SQLite 数据库所在目录和 Skills 根目录。
- 管理 Workbench Skills 根目录。
- 查看支持的工具及其全局 Skills 目录。
- 调整 Skills 列表中全局工具的展示顺序。
- 新增、编辑和删除自定义 Agent 工具。
- 在支持的工具目录列表中展示工具图标和内置/自定义类型。
- 打开不存在的工具 Skills 目录时，先确认是否创建对应目录。
- 配置关闭窗口时的行为：隐藏到托盘或退出应用；默认隐藏到托盘，并在首次关闭窗口时提示可从托盘恢复或退出。
- 展示 Skills 路径映射关系。
- 管理主题偏好。

## 6. 页面结构

Workbench 采用左侧模块导航 + 右侧工作区。

左侧导航当前包含：

- 项目
- Skills
- 资源 Radar
- 设置

右侧工作区根据当前模块显示列表、详情、表单或状态面板。

后续可以在不破坏模块主入口的前提下增加 Dashboard、项目工作台、工具状态视图或自动化入口。

## 7. 关键交互

### 7.1 添加项目

用户选择或输入本地项目路径，Workbench 生成项目记录。用户可以补充备注、标签和启动配置。

成功标准：

- 项目出现在项目列表中。
- 项目路径可以被打开。
- 项目详情可以查看和编辑基础信息。
- 项目记录可以删除，且不会影响本地项目目录。
- GitHub/Gitee 导入会在 clone 前识别项目记录和目标目录的四种组合状态，并且不会覆盖已有目录。
- 配置启动项后，可以通过启动按钮在 Workbench 内为所有启用启动项创建本次启动会话。
- 项目本次启动输出可以进入启动日志详情页查看，并可在全部输出和单个启动项输出之间切换。

### 7.2 管理和启用 Skills

用户配置统一 Skills 根目录后，Workbench 扫描其中的 `SKILL.md` 并展示 Skills。用户可以为 Skill 设置分类，并选择需要全局启用或在指定项目中启用的工具。

成功标准：

- 能展示扫描到的 Skill。
- 能搜索 Skill。
- 能打开统一 Skills 根目录。
- 能打开 Skill 文件。
- 能按分类管理和筛选 Skills。
- 能通过 Auto 同步在支持的全局工具目录中启用或停用 Skill。
- 能通过 Auto 同步在支持的项目工具目录中启用或停用 Skill。
- Workbench Skills 根目录始终保存唯一真实副本。
- 目标冲突时不会自动覆盖、合并或删除已有内容。
- 全局工具启用状态可以被识别为未启用、Workbench 管理或内容冲突。
- 内容冲突只能通过用户明确选择唯一版本源解决，且替换前会备份被替换版本。
- 能删除统一根目录中的 Skill，并清理 Workbench 管理的启用记录和受管目标。

### 7.3 导入 Skills

用户选择 ZIP 文件或已解压文件夹作为导入来源。Workbench 扫描其中所有包含 `SKILL.md` 的目录，并批量导入所有可导入 Skills。

成功标准：

- 能通过系统选择器选择 ZIP 文件或已解压文件夹。
- 能将来源中的多个 Skills 一次性复制到统一 Skills 根目录。
- 导入完成后 Skill 默认不启用。
- 同名 Skill 不覆盖、不合并，并明确提示冲突。
- 导入结果展示成功、跳过、无效数量，并支持打开统一根目录。

### 7.3.1 skills.sh 技能市场

用户可以在 Skills 模块内切换到 `技能市场` 浏览 `skills.sh` 条目，并由 Workbench 安装或卸载 GitHub 来源 Skill 到统一 Skills 根目录。用户也可以在 `更新` 子视图检查和更新由 Workbench 从 `skills.sh` 安装的 Skill。

成功标准：

- 能在 Skills 模块内切换 `本地 Skills`、`技能市场` 和 `更新` 子视图。
- 能浏览、搜索并查看 `skills.sh` Skill 详情。
- 能按安装状态筛选市场条目，并看到全部、已安装、未安装、可更新、不支持和当前结果数量。
- 能安装 GitHub 来源 Skill 到 Workbench 统一 Skills 根目录。
- 已安装的市场 Skill 操作列显示“卸载”，卸载前必须确认。
- 市场初次加载使用列表与详情骨架屏，加载失败时提供明确错误和重试入口。
- 安装时以后端阶段事件反馈百分比进度，且同一时间只执行一个市场安装操作。
- 市场下载必须有连接和读取超时，并限制异常大型仓库占用本地资源。
- 市场列表在当前应用进程内缓存；切换子视图不重复加载，手动刷新和安装完成后重新获取最新状态。
- 安装完成后 Skill 默认不自动启用到任何 Agent 工具目录。
- 非 GitHub 来源 Skill 明确展示为 Workbench 自管安装暂不支持。
- 从市场安装的 Skill 会记录来源元数据，用于后续更新检查；卸载或删除该 Skill 时清理来源元数据。
- 更新页只管理 Workbench 从 `skills.sh` 安装的 Skill。
- 能检查更新、单项更新、选中批量更新和更新全部可更新项。
- 更新页复选框只允许选择状态为可更新的 Skill；已是最新、检查失败或正在更新的条目不可勾选。
- 更新前备份统一根目录中的旧版本；单项失败不应删除旧版本。
- 更新结果按 Skill 逐项展示；批量任务中的单项失败不应阻止其他项继续执行。

### 7.4 管理资源 Radar 条目

用户可以手动添加资源 Radar 条目或同步 GitHub Stars，并通过搜索、分类、领域、来源、语言、收藏、来源状态和重复状态进行管理。搜索会匹配用户标签和 GitHub Topics。

成功标准：

- 条目可新增、编辑、删除、查看、搜索和筛选。
- 收藏状态可切换。
- 链接可打开。
- GitHub Stars 重复同步不会创建重复资源。
- GitHub Stars URL 唯一匹配手动资源时合并来源，保留手动资源的分类、领域、标签、备注和收藏状态。
- 多条手动资源匹配同一个 GitHub repo 时进入待合并重复组，由用户选择主资源。
- 合并后来源合并、标签去重、备注追加、收藏取 OR，主资源保留分类和领域，副资源删除。
- 同步不覆盖用户维护的分类、标签、备注和收藏状态。
- 取消 Star 只标记来源失效，不删除本地资源。

## 8. 当前非目标

以下内容不是当前已实现能力，进入开发前需要单独确认范围、数据所有权、风险和验收标准：

- 交互式终端、PTY、SSH、终端标签页或持久化启动日志。
- Agent 配置、prompts、MCP 配置或规则文件管理。
- 从在线来源安装 Skills。
- Skills 在线市场、远程仓库管理和自动更新。
- 资源 Radar 后台采集、定时同步或 LLM 分类。
- Obsidian 连接或笔记管理。
- 插件系统。
- 多用户、登录、云同步。
- 合并参考项目代码。
- 将参考项目组成 monorepo。

这些方向可以进入后续路线图，但不应在没有计划和验证边界时直接实现。

## 9. 技术方向

当前采用：

- Tauri
- React
- TypeScript
- SQLite

应用包含前端界面、本地 Tauri 后端能力和本地数据库。当前不启动独立 HTTP 后端服务。

本地数据建议放在系统标准 app data 目录中，源码目录只保存项目代码和文档。

## 10. 当前质量门槛

当前版本应持续满足：

- 可以启动 Workbench 桌面应用。
- 左侧导航和基础页面结构可用。
- 项目、Skills、资源 Radar 和设置模块可使用真实本地数据。
- 项目新增、编辑、删除和启动配置可持久化保存。
- 项目启动会话可创建、查看和停止。
- Skills 统一根目录可配置、扫描和打开。
- Skill 分类、筛选、导入、启用、停用、冲突解决和删除行为可验证。
- 删除或停用 Skill 不会误删未被 Workbench 管理的工具目录内容。
- 资源 Radar 可增删改查、搜索筛选、收藏、打开链接和同步 GitHub Stars。
- 数据在应用重启后仍然保留。
- `pnpm verify` 通过。
- `pnpm tauri:verify-build` 在发布前通过。

## 11. 后续方向

进阶开发可以从以下方向逐步展开。每一项进入实现前都应先形成计划，明确范围、非目标、数据模型、风险和验证方式。

### 11.1 项目工作台

- 项目详情内聚合启动项、启动日志、项目 Skills、项目资源、打开方式和常用操作。
- 支持单独启动某个启动项。
- 支持持久化启动历史和失败摘要。
- 支持项目模板或项目初始化 checklist。
- 支持项目级备注、资源、命令和 Skills 的一体化视图。

### 11.2 Skills 进阶管理

- 工具真实状态视图：按 Codex、Claude Code、OpenCode 或自定义工具查看实际 Skills 目录状态。
- 项目详情的“添加 Skills”弹层：从统一库批量选择 Skills 并指定项目内工具。
- Skill 详情内预览 `SKILL.md` 和相关 README。
- Copy 副本显式重新同步。
- 批量启用、停用、设置分类和导出。
- 更清晰的冲突对比与解决预览。
- 内置工具路径覆盖。
- 技能组或项目模板，用于保存一组常用 Skills。
- 在线来源导入：Git URL、压缩包 URL、skills.sh 或其他 Skills 市场。
- Git 来源更新检查、差异预览和手动更新。
- Skills 备份、恢复和版本历史。

### 11.3 Agent 工具与配置

- 管理 Agent 配置、prompts、MCP 配置、规则文件和环境变量。
- 展示各 Agent 的安装状态、配置路径和可用能力。
- 为项目绑定推荐 Agent、启动命令和 Skills 组合。
- 提供 Agent 环境诊断，例如 PATH、认证状态、配置文件冲突。

### 11.4 资源 Radar 进阶

- 支持 JSON、Markdown、CSV 等导入来源。
- 支持 radar inbox，由外部 collector 或 Codex automation 写入候选资源。
- 支持更多外部来源，例如 GitHub trending、论文源、RSS、浏览器导出或本地文件夹。
- 支持重复检测、来源健康、收藏分组和阅读状态。
- 支持可选的 LLM 摘要、标签建议和分类建议，但用户最终确认。
- 支持资源与项目、Skills、Agent 配置的关联。

### 11.5 自动化与任务入口

- 创建本地自动化任务入口，统一管理手动触发、定时触发和外部触发。
- 支持项目启动前后 hook。
- 支持资源同步、Skills 检查、环境诊断等可重复任务。
- 支持操作日志、失败重试和导出诊断包。

### 11.6 知识库和文档连接

- Obsidian 知识库连接。
- 将项目、资源、Skills 和开发记录导出为 Markdown。
- 将 Workbench 中的操作经验沉淀为文档或 VibeCoding 笔记。

### 11.7 数据安全与同步

- 操作日志和日志导出。
- 本地备份和恢复。
- 可选 Git 备份。
- 可选多设备同步。
- 敏感配置加密保存。

### 11.8 平台化能力

- 插件或扩展机制。
- 自定义面板和自定义数据源。
- 第三方工具连接。
- 多用户或团队协作。

平台化能力属于远期方向，只有当本地个人工作台体验稳定后再考虑。
