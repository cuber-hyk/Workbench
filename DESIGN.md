# Workbench App 设计系统

## Authority And Scope

本文档记录 Workbench App 已确认的 UI 方向，是设计意图、布局规则、可复用 UI 模式和禁用模式的来源。

精确视觉值保存在 `design-tokens.json`。当前代表性视觉来源是 `UI/` 中的静态原型。

## 2. 已确认的产品 UI 方向

Workbench App 使用本地优先的桌面工作台界面。整体体验应接近实用型开发控制台：安静、紧凑、清晰，适合日常高频使用。

Workbench App 不是营销站、落地页、知识图谱或装饰性 Dashboard。首屏应直接呈现真实可用的工作台模块。

## 3. App Shell

Workbench 使用固定左侧导航 + 右侧工作区。

已确认导航模块：

- 项目
- Skills
- 资源 Radar
- 设置

规则：

- 左侧边栏是持久模块切换入口。
- 右侧工作区根据当前模块切换内容。
- MVP 不设置独立 Dashboard 模块。
- 当有助于表达本地工作台定位时，侧边栏可以展示本机工作区状态。
- 主题切换属于 App Shell 和设置模块。

来源：

- `docs/PRD.md`
- `docs/ARCHITECTURE.md`
- `UI/index.html`

## 4. 布局模式

### 4.1 模块页面

每个主模块页面使用：

- 页面标题区，包含标题、简短上下文说明和必要的主操作。
- 工具栏，用于搜索和筛选。
- 主内容区。

高度规则：

- App Shell 使用固定 `100dvh` 高度。
- 页面外层不产生滚动条。
- 列表区、详情区和设置页主体是明确的滚动容器。
- 主内容区使用 flex 和 `min-height: 0` 控制高度，不依赖硬编码 `calc(100vh - ...)`。

### 4.2 列表 + 详情

项目、Skills 和资源 Radar 视图使用列表 + 详情的工作模式。

规则：

- 列表是模块内部的主要导航区域。
- 详情面板展示当前选中项及其相关操作。
- 编辑可以在详情面板中以表单字段呈现，也可以使用聚焦型弹窗。
- 搜索和筛选位于列表上方，不放在详情面板内部。

### 4.3 设置

设置页面使用纵向堆叠面板。每个面板对应一个配置主题，例如 Skills 存储、支持的工具目录、本地数据或主题。

设置页路径行使用左侧名称和路径、右侧操作与状态聚合的布局。路径输入旁的打开目录图标必须和输入框同高同线。

## Component Rules

### 5.1 按钮

按钮类型：

- 默认按钮：用于次要操作。
- 主按钮：用于页面或弹窗的主要操作。
- 图标按钮：用于编辑、关闭等紧凑操作。
- 全宽按钮：用于详情面板中明确的局部主操作。

规则：

- 按钮高度保持紧凑。
- 操作使用图标或短标签。
- 不使用营销式的大型 CTA 按钮。

### 5.1.1 路径输入

路径输入保留可手动编辑的文本框，并在同一行右侧提供文件夹图标按钮打开系统目录选择器。

规则：

- 项目路径和启动工作目录使用相同的输入 + 图标按钮布局。
- 文件夹图标按钮使用固定宽度，不挤压路径输入框。
- 选择目录后直接回填对应字段；取消选择不改变现有内容。

### 5.2 Row Card

Row Card 用于项目和 Radar 列表项。

规则：

- Row Card 展示标题、状态或分类元信息，以及可选标签或简短备注。
- 选中态和悬停态使用同一语义强调色系。
- Row Card 是可点击列表项，不是装饰性内容卡片。

### 5.3 Table Row

当模块需要横向比较多项属性时使用 Table Row，例如项目的标签、启动项和启动状态，以及 Skills 的分类、全局启用状态和项目启用状态。

规则：

- 在滚动列表中，表头可保持 sticky。
- 单元格内容应安全截断或换行，不能撑坏布局。
- 工具启用标识保持紧凑。
- 项目列表使用表格展示项目、标签、启动项、启动状态和操作。
- 项目列表中的打开目录、启动和编辑属于行级高频操作，应放在操作列。
- 项目列表行级操作在默认窗口宽度下必须完整可见；空间不足时优先压缩或隐藏低优先级信息列，不隐藏操作列。
- 项目列表启动状态只表达项目级摘要：未配置、可启动、运行中、部分运行、失败、已停止、已结束。
- 项目归档状态使用紧凑标签表达；归档和恢复属于行级操作，可放在项目表格操作列。
- 项目启动主操作执行所有启用启动项；本次启动会话支持停止或重新启动单个启动项。
- Skills 列表中的全局工具标识可以直接点击切换启用状态；内容冲突状态只展示状态，不直接切换。
- 全局工具标识使用紧凑工具图标，图标形态参考 cc-switch 的 Codex、Claude、OpenCode 工具入口。
- Skills 列表中的分类标签支持双击编辑，空值保存为“未分类”。
- Skills 列表中的操作列承载打开 `SKILL.md` 和删除图标按钮；删除点击后必须弹出确认弹窗。
- 资源 Radar 使用 Row Card 列表；收藏、打开链接和编辑集中在详情底部操作栏左侧，删除位于右侧且删除前必须弹出确认弹窗。
- 资源 Radar Row Card 元信息展示资源类型、领域、来源、语言、Stars 数量和更新时间；收藏使用右侧星标图标，不在元信息里写“已收藏”。
- 资源 Radar 的来源、语言、Stars 数量、来源失效状态和待合并重复状态使用文字明确展示，不能仅依赖颜色。
- 资源 Radar 顶部工具栏默认只展示搜索、类型、领域、来源、更多筛选和收藏星标，避免默认窗口下筛选项铺满两行。
- 资源 Radar 的语言、来源状态和重复状态放入“更多筛选”；标签和 GitHub Topics 只参与搜索，不提供独立下拉筛选。
- 资源 Radar 收藏星标使用 attention 色表达已收藏，未收藏使用 muted 色。
- Skills 操作列宽度使用 `skillsActionColumnWidth` token，避免按钮换行。

### 5.4 详情面板

详情面板用于呈现选中项。

规则：

- 顶部显示主名称和次级元信息。
- 详情使用表单式字段或小型标签块。
- 文件路径和命令使用等宽字体，并且必须安全换行。
- 重要边界说明使用紧凑提示块。
- Skill 出现多个全局版本冲突时，使用 Skill 级冲突面板统一选择唯一版本源。
- 冲突面板必须先说明状态，再展示 `.workbench`、`.codex`、`.claude`、`.opencode` 版本选项，最后展示应用和打开目录等辅助操作。
- 资源 Radar 出现 GitHub Stars 重复组时，在列表上方使用紧凑冲突面板展示候选资源；用户直接选择主资源并执行合并，不使用额外弹窗。
- 资源 Radar 详情标题右侧展示可点击收藏星标；详情底部常用操作与危险操作分行，避免默认窗口宽度下按钮挤压。
- 全局工具启用只在列表表格中操作，详情面板不重复展示全局工具启用区。
- 项目启用区按项目提供总开关，批量控制该项目下当前 Skill 的所有工具启用状态。
- 打开文件和删除 Skill 属于行级操作，放在表格操作列；详情面板只展示路径信息，不重复放操作按钮。
- 项目详情面板展示项目字段、备注、启动配置、最近启动时间和本次启动会话摘要；打开目录和启动项目不在详情面板重复展示。
- 本次启动会话状态来自当前内存中的启动会话组，不代表历史运行记录。
- 项目详情面板不展示完整启动日志；启动摘要提供查看日志入口，进入项目模块内的日志详情页。
- 启动日志详情页使用顶部返回、启动组元信息、会话 Tabs 和大面积等宽日志区域；不脱离现有 App Shell，不新增主导航。
- 启动日志区域占满详情页剩余空间，使用紧凑 Tabs 顶栏和黑底高密度日志排版；日志中的 HTTP / HTTPS 地址可点击并交由系统浏览器打开。

### 5.5 弹窗

弹窗用于创建、导入等聚焦任务。

规则：

- 弹窗只承载当前任务。
- 导入来源选择优先使用系统文件选择器或文件夹选择器。
- 导入弹窗展示导入摘要、状态列表、导入边界提示和最终操作。
- 导入完成后可以提供打开统一根目录入口。
- 弹窗头部和底部操作区保持固定，只有内容区滚动。
- 破坏性或冲突相关信息必须在确认前可见。

### 5.6 主题

Workbench 支持浅色和深色主题。

规则：

- App Shell 和设置中的主题状态应保持一致。
- 两种主题使用相同布局和组件层级。
- 深色主题不是另一套视觉方案，而是同一工作台规则在深色 token 下的表达。

## 6. 文案规则

规则：

- UI 默认使用中文。
- 技术标识、路径、命令、API 名称和工具名称可以保留英文。
- 页面文案应简洁、操作导向。
- 避免用可见文案解释显而易见的 UI 机制。
- 当文案描述重要行为边界时，可以使用边界提示，例如启动会话日志只保存在当前内存中。

## Foundations

已确认基础：

- 紧凑的桌面优先布局。
- 克制的中性色界面，以 teal 作为主强调色。
- orange 仅用于收藏或注意状态。
- 使用边框区分面板和列表行。
- 圆角保持较小，通常为 4px 到 8px。
- 字体使用系统 UI 字体，并包含 Microsoft YaHei fallback。
- 路径、命令和代码类值使用等宽字体。

禁用模式：

- 营销式 hero 布局。
- 装饰性渐变光斑或虚化背景。
- 绕过 App Shell 的一次性页面风格。
- 普通页面区域中嵌套卡片。
- 依赖 AI 生成装饰图的 UI。

## 8. 可访问性和交互规则

规则：

- 可交互列表行必须有明确的悬停态和选中态。
- 按钮和输入框在浅色、深色主题下都必须保持可读对比度。
- 主按钮文字使用对应主题的 `accentContrast` token，不能假设强调色始终适合白色文字。
- 文本不能溢出容器；路径和命令必须有明确的换行或截断策略。
- 将静态原型迁移为 React 组件时，必须考虑键盘和焦点行为。

## 9. 暂定规则

以下规则已经出现在当前原型中，但应在正式实现时继续确认：

- 最小桌面宽度从 1024px 开始。
- 列表 + 详情页面右侧详情面板宽度约为 396px。
- 项目和 Skills 列表使用表格行，Radar 列表使用 Row Card。
- 侧边栏宽度为 220px。
- Skills 表格行最小高度、全局工具列宽和操作列宽应使用 `design-tokens.json` 中的表格相关 token。

## Known Gaps

以下场景尚未设计：

- 移动端布局。
- 长时间运行的导入进度。
- 可访问性焦点环细节。
- 最终 React 共享组件 API。
