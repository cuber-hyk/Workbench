# Workbench App 设计系统

## Authority And Scope

本文档记录 Workbench App 已确认的 UI 方向，是设计意图、布局规则、可复用 UI 模式和禁用模式的来源。

精确视觉值保存在 `design-tokens.json`。当前代表性视觉来源是 `UI/` 中的静态原型。

## 2. 已确认的产品 UI 方向

Workbench App 使用本地优先的桌面工作台界面。整体体验应接近实用型开发控制台：安静、紧凑、清晰，适合日常高频使用。

Workbench App 不是营销站、落地页、知识图谱或装饰性 Dashboard。首屏应直接呈现真实可用的工作台模块。

## 3. App Shell

Workbench 使用固定左侧导航 + 右侧工作区。

已确认导航模块：

- 项目
- Skills
- AI Radar
- 设置

规则：

- 左侧边栏是持久模块切换入口。
- 右侧工作区根据当前模块切换内容。
- MVP 不设置独立 Dashboard 模块。
- 当有助于表达本地工作台定位时，侧边栏可以展示本机工作区状态。
- 主题切换属于 App Shell 和设置模块。

来源：

- `docs/PRD.md`
- `docs/ARCHITECTURE.md`
- `UI/index.html`

## 4. 布局模式

### 4.1 模块页面

每个主模块页面使用：

- 页面标题区，包含标题、简短上下文说明和必要的主操作。
- 工具栏，用于搜索和筛选。
- 主内容区。

### 4.2 列表 + 详情

项目、Skills 和 AI Radar 视图使用列表 + 详情的工作模式。

规则：

- 列表是模块内部的主要导航区域。
- 详情面板展示当前选中项及其相关操作。
- 编辑可以在详情面板中以表单字段呈现，也可以使用聚焦型弹窗。
- 搜索和筛选位于列表上方，不放在详情面板内部。

### 4.3 设置

设置页面使用纵向堆叠面板。每个面板对应一个配置主题，例如 Skills 存储、支持的工具目录、本地数据或主题。

## Component Rules

### 5.1 按钮

按钮类型：

- 默认按钮：用于次要操作。
- 主按钮：用于页面或弹窗的主要操作。
- 图标按钮：用于编辑、关闭等紧凑操作。
- 全宽按钮：用于详情面板中明确的局部主操作。

规则：

- 按钮高度保持紧凑。
- 操作使用图标或短标签。
- 不使用营销式的大型 CTA 按钮。

### 5.2 Row Card

Row Card 用于项目和 Radar 列表项。

规则：

- Row Card 展示标题、状态或分类元信息，以及可选标签或简短备注。
- 选中态和悬停态使用同一语义强调色系。
- Row Card 是可点击列表项，不是装饰性内容卡片。

### 5.3 Table Row

当模块需要横向比较多项属性时使用 Table Row，例如 Skills 的分类、全局启用状态和项目启用状态。

规则：

- 在滚动列表中，表头可保持 sticky。
- 单元格内容应安全截断或换行，不能撑坏布局。
- 工具启用标识保持紧凑。

### 5.4 详情面板

详情面板用于呈现选中项。

规则：

- 顶部显示主名称和次级元信息。
- 详情使用表单式字段或小型标签块。
- 文件路径和命令使用等宽字体，并且必须安全换行。
- 重要边界说明使用紧凑提示块。

### 5.5 弹窗

弹窗用于创建、导入等聚焦任务。

规则：

- 弹窗只承载当前任务。
- 导入弹窗可以包含来源选择、扫描结果、冲突提示和最终操作。
- 破坏性或冲突相关信息必须在确认前可见。

### 5.6 主题

Workbench 支持浅色和深色主题。

规则：

- App Shell 和设置中的主题状态应保持一致。
- 两种主题使用相同布局和组件层级。
- 深色主题不是另一套视觉方案，而是同一工作台规则在深色 token 下的表达。

## 6. 文案规则

规则：

- UI 默认使用中文。
- 技术标识、路径、命令、API 名称和工具名称可以保留英文。
- 页面文案应简洁、操作导向。
- 避免用可见文案解释显而易见的 UI 机制。
- 当文案描述重要 MVP 行为边界时，可以使用边界提示，例如项目启动不捕获日志。

## Foundations

已确认基础：

- 紧凑的桌面优先布局。
- 克制的中性色界面，以 teal 作为主强调色。
- orange 仅用于收藏或注意状态。
- 使用边框区分面板和列表行。
- 圆角保持较小，通常为 4px 到 8px。
- 字体使用系统 UI 字体，并包含 Microsoft YaHei fallback。
- 路径、命令和代码类值使用等宽字体。

禁用模式：

- 营销式 hero 布局。
- 装饰性渐变光斑或虚化背景。
- 绕过 App Shell 的一次性页面风格。
- 普通页面区域中嵌套卡片。
- 依赖 AI 生成装饰图的 UI。

## 8. 可访问性和交互规则

规则：

- 可交互列表行必须有明确的悬停态和选中态。
- 按钮和输入框在浅色、深色主题下都必须保持可读对比度。
- 文本不能溢出容器；路径和命令必须有明确的换行或截断策略。
- 将静态原型迁移为 React 组件时，必须考虑键盘和焦点行为。

## 9. 暂定规则

以下规则已经出现在当前原型中，但应在正式实现时继续确认：

- 最小桌面宽度从 1024px 开始。
- 列表 + 详情页面右侧详情面板宽度约为 396px。
- Skills 列表使用表格行，项目和 Radar 列表使用 Row Card。
- 侧边栏宽度为 220px。

## Known Gaps

以下场景尚未设计：

- 移动端布局。
- 空状态。
- 加载状态。
- 简单提示之外的错误状态。
- 真实表单校验状态。
- 长时间运行的启动或导入进度。
- 可访问性焦点环细节。
- 最终 React 共享组件 API。
